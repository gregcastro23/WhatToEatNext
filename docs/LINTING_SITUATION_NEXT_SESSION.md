**🎯 ROOT CAUSE IDENTIFIED (2026-07-05, wave4):** the phantom-diff incident below, and likely the stash-sweep incidents above too, were substantially **self-inflicted by this campaign's own workflow design**, not (only) an external process. When a workflow runs many `agent()` calls concurrently against the same shared working directory with no isolation, and those agents run their own ad-hoc `git stash` / `git checkout <file>` for self-verification ("let me stash to check the pre-edit baseline," "let me checkout to test before/after"), **`git stash` operates on the entire working tree, not per-file** — it silently sweeps away every OTHER concurrently-running sibling agent's in-progress, uncommitted edit at that instant. A wave4 reviewer for `refinedGrains.ts` caught this red-handed: it ran `git stash` itself mid-review, which reverted its own target file back to HEAD *and* wiped `prometheus-metrics.ts`'s real edit from a sibling agent — which is exactly the file wave4's OWN reviewer for `prometheus-metrics.ts` then flagged as "fabricated," when in fact the implementer's original report had almost certainly been accurate at the time it was written. **Fix applied starting wave4**: agent prompts now explicitly forbid `git stash`/`git checkout`/`git reset` for self-verification — read-only commands only (`git diff`, `git status`, `git show`, `git log`), plus an instruction to base the final report on a `git diff` run as the literal last action. Combined with smaller wave sizes (9 files, ~10 min wall-clock vs. wave3's 30 files/~3 hours), this took the phantom rate from 16/21 (76%) down to 1/7 (14%) in the very next wave — strong evidence this was the dominant cause. **Any future wave design must never let agents run stateful git commands (stash/checkout/reset) against the shared working tree** — only this top-level session should ever do that, and only between waves when no other agent is active.
